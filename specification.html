1. Scope and authority

Code defines the protocol boundary.

This page describes the Atomicals ARC-20 fungible-token model on Bitcoin. It is a compatibility guide, not an independent consensus specification, smart-contract specification, or financial statement.

Authority order

The Atomicals specification guide states that the specification is defined in code. The selected validator or indexer determines version-sensitive behavior.

Do not infer validity from presentation

A ticker, image, metadata object, portfolio label, or generic Bitcoin confirmation cannot establish token validity or ownership by itself.

  • A production integration MUST pin a code revision or release and its activation configuration.
  • A builder MUST generate the envelope and allocation expected by that exact target.
  • This document covers ARC-20 fungible tokens only. It does not establish an ARC-721 standard.

Keep identifiers, units, and UTXOs distinct.

ARC-20 safety starts with precise vocabulary. A displayed ticker and a selected satoshi are not interchangeable forms of identity or value.

3. Envelope and identity

An ARC-20 UI form is not the wire format.

Mint and update workflows use Taproot commit and reveal mechanics. The reveal contains a non-executed Atomicals envelope whose operation and CBOR payload are validated in context.

Conceptual Atomicals envelope

  "atom"
  <operation>
  <CBOR payload>
OP_ENDIF

Ticker handling

Always retain the Atomical ID.

An integration MUST keep an Atomical ID next to any displayed ticker and SHOULD record source transaction, mint mode, source indexer, and validation version.

The pinned validator's direct-FT ticker pattern, lowercase [a-z0-9]{1,21}, is a revision fact. Validate names against the active target, not a display-only rule.

Authenticity is outside ticker availability

Show users the resolved Atomical ID and warn when visual metadata, source history, or mint configuration does not match the intended asset.

4. Issuance

Direct FT, fixed DFT, and perpetual DFT are different paths.

Supply behavior comes from the selected operation, output values, deployment configuration, and active implementation rules. Never collapse these modes into a generic mint form.

Direct FT from the Atomicals guide
yarn cli mint-ft <tick> <total_supply> metadata.json

Direct FT is the ft operation. Under the pinned validator, full supply equals output 0's satoshi value. A 100,000,000-unit mint needs a 100,000,000-satoshi output.

It is a colored-satoshi issuance model, not collateral, redemption, a price floor, stablecoin peg, or financial promise.

Fixed DFT from the Atomicals guide
yarn cli init-dft <tick> <per_mint_amt> \
  <mint_count> <start_height> metadata.json

yarn cli mint-dft <tick>

Nominal maximum issuance is mint_amount * max_mints. A dmt claim follows the deployment rather than a free-form transfer amount.

DFT claim

Output 0 is exact.

For the pinned validator, a valid dmt claim must meet deployment height, quota, Bitwork and activation constraints, then place exactly mint_amount satoshis in output 0.

Perpetual DFT

Do not estimate it from a large fixed cap.

It is separate activation-gated behavior with its own progression and Bitwork-vector rules. Read the actual deployment and target version before showing a supply cap.

5. Metadata

Display data is optional and untrusted.

The ARC-20 guide describes metadata as any valid JSON object. It can assist presentation but never establish protocol truth, identity, or financial rights.

  • name, desc, image, decimals, links, and legal are common display fields, not validation inputs.
  • decimals is presentation metadata. It MUST NOT create sub-satoshi ARC-20 units.
  • URLs, media, names, legal text, and social links MUST NOT establish authenticity, ownership, financial rights, or legal validity.
  • Renderers SHOULD sanitize metadata and URLs before display.

6. Transfer allocation

Transfers are a concrete transaction-allocation problem.

ARC-20 is not an account debit plus credit. Input order, output order, exact output values, activation state, and advanced payloads can all change the allocation result.

Evaluate outputs in order

For normal FT coloring in the pinned reference builder, eligible transaction outputs are considered sequentially.

Require the whole output value to fit

An output receives colored units only while its full BTC satoshi value fits inside the remaining colored amount.

Record any unfit remainder

If the next eligible output is too large, the remainder is not automatically assigned and can be recorded as burned.

Reject inflation

Colored output totals cannot exceed the corresponding colored input total.

CaseColored inputOutputsExpected result under pinned normal allocation
Clean split1,200 sats / 1,200 unitsoutput 0: 700 sats; output 1: 500 sats700 units then 500 units. No colored remainder.
Oversized next output1,200 sats / 1,200 unitsoutput 0: 700 sats; output 1: 546 satsOutput 0 receives 700 units. The remaining 500 units do not fit the next output and are burned.

Advanced operations need active-version validation

The pinned code recognizes y for FT split behavior and activation-gated z custom coloring. Do not handcraft their implementation-specific payloads from an amount field. The x splat operation belongs to the NFT allocation branch, not a generic ARC-20 transfer path.

7. Security and product scope

Fail closed before, during, and after a signature.

Wallet isolation, transparent output previews, and post-broadcast reconciliation are core integration requirements, not polish.

Wallet and transaction requirements

Protect colored outpoints.

  • Keep colored Atomicals inputs separate from cardinal fee inputs.
  • Show output order, BTC value, expected colored value, recipient, retained value, and expected burns before signature.
  • Verify allocation after broadcast and appropriate confirmations.
  • Never request seed phrases, private keys, or wallet exports to verify ARC-20 state.

Universe compatibility boundary

Do not present unavailable product behavior as supported.

  • Direct mint-ft issuance is not exposed by current Universe flows.
  • Visible ARC-20 transfer, partial transfer, split, and combine are not safe to use or document as supported.
  • Discovery and portfolio are ElectrumX proxy-backed convenience views, not canonical proof.
