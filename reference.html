1. Envelope and identity

Atomicals operations are carried in a Taproot reveal.

The Atomicals guide describes a commit and reveal construction. The reveal witness script carries a non-executed envelope marked with atom, an operation code, and a CBOR payload. The guide says its actual specification is defined in code.

Conceptual Atomicals envelope
OP_FALSE
OP_IF
  "atom"
  <operation>
  <CBOR payload>
OP_ENDIF

Important boundary

This is not a JSON inscription protocol.

Atomicals uses CBOR inside a Taproot witness-script envelope. A UI may collect familiar fields, but a builder must serialize the exact operation and payload the active validator expects.

Atomical ID

Use the transaction-based identifier <txid>i<output-index> to identify an Atomical. Keep it beside a display ticker in your data model and verification screens.

Ticker is a locator, not proof of authenticity

The ARC-20 guide describes globally unique ticker registration and says the claim becomes available after three confirmations. Verify ticker resolution, Atomical ID, source transaction, mint configuration, and current history. Do not validate a project from a ticker, image, metadata link, or exchange label alone.

3. Issuance reference

Direct FT and DFT answer different supply questions.

Direct mint puts all units in one colored output. Fixed DFT sets a rules-based mint campaign. Perpetual DFT is a separate, activation-dependent variant and must be evaluated from its own configuration.

Direct FT CLI form from the Atomicals guide
yarn cli mint-ft <tick> <total_supply> metadata.json

Illustration from the guide: a 100,000,000-unit direct mint has a 100,000,000-satoshi output, or 1 BTC. This is an issuance and allocation fact, not a price guarantee, redeemability promise, or collateral claim.

Fixed DFT CLI forms from the Atomicals guide
yarn cli init-dft <tick> <per_mint_amt> \
  <mint_count> <start_height> metadata.json

yarn cli mint-dft <tick>

For a fixed deployment, nominal maximum issuance is per_mint_amt * mint_count. A claim is valid only when its deployment rules, timing, amount, and current quota are satisfied.

Before mint

Check availability and context

Confirm the exact ticker state with a capable indexer. A pending or competing claim is not a safe basis for issuing or branding a token.

At deployment

Preserve the configuration

Store the deployment Atomical ID, exact start height, mint amount, maximum mints, optional Bitwork rule, metadata digest, and source revision used to build it.

At claim

Check output 0 precisely

The pinned validator requires a DMT's output 0 to equal the deployment's mint amount. Do not substitute a generic amount entry for this transaction rule.

4. Metadata

Display data is optional and untrusted.

The ARC-20 guide describes metadata as any valid JSON object and recommends informative fields. Treat it as content for presentation, not authority for supply, legitimacy, fractional units, legal rights, or project identity.

Guide-shaped metadata example
{
  "name": "Example",
  "desc": "Display text",
  "image": "atom:btc:dat:<file>/image.png",
  "decimals": 0,
  "links": {},
  "legal": { "terms": "" }
}
name, desc, image
Common display conventions. Sanitise rendering and do not infer provenance from a URL or media asset.
decimals
Presentation metadata. ARC-20 native quantities remain integer satoshi units, so this field does not authorize sub-satoshi fractions.
links, legal
Optional assertions supplied by the creator. Validate terms, branding, identity, and jurisdictional claims outside the protocol.

5. Transfers

Normal coloring is sequential and value-sensitive.

For the pinned reference implementation, normal FT allocation considers eligible outputs in order. An output receives units only when its full BTC value fits into the remaining colored amount. If it does not fit, the remaining units can be recorded as burned.

Clean allocation illustration

1,200 colored units in, 1,200 colored units out

input: 1,200 sats / 1,200 units

output 0: 700 sats / 700 units to recipient

output 1: 500 sats / 500 units to sender

Both output values fit the sequential remainder, so the full 1,200-unit amount can be allocated. Fee inputs and outputs are omitted here for clarity.

Burn-risk illustration

An oversized next output breaks the remainder

input: 1,200 sats / 1,200 units

output 0: 700 sats / 700 units

output 1: 546 sats / 0 units

After output 0, only 500 units remain. Because the next 546-sat output is too large for the remainder under normal allocation, the remaining 500 units are not automatically protected by a generic change output.

Never derive allocation from a send amount alone

Inputs, their ordering, every output's BTC value and position, activation set, and any advanced coloring operation determine the result. Ask the active indexer to decode or validate the exact PSBT or transaction blueprint before broadcast, then verify its post-broadcast report.

No inflation

Colored output total cannot exceed colored input total.

The pinned validator rejects allocation that would create more units than are present in the relevant colored inputs. It can still record burns when the transaction does not allocate a remainder cleanly.

Advanced allocation

Split and custom coloring require version checks.

Use y or activation-gated z only through software that validates the precise payload and network rules. Do not reconstruct their behavior from this overview.

7. Preflight

Minimum evidence before a signature.

A production builder should turn these checks into testable constraints, retain the decoded blueprint with the user intent, and fail closed when an indexer cannot report a safe allocation.

  • Resolve the ticker to the expected Atomical ID and inspect the mint configuration, not only its display metadata.
  • Confirm the network, activation set, indexer version, and source revision used to generate the preview.
  • List colored inputs separately from cardinal fee inputs, then show their order and exact sat values.
  • Show every output's position, BTC value, expected colored quantity, recipient, and whether any colored quantity is burned.
  • For DFT claims, verify start height, remaining mint quota, required Bitwork, and exact output 0 amount.
  • After broadcast, wait for the required confirmation and indexer recognition before treating a ticker claim, mint, or balance as final.
